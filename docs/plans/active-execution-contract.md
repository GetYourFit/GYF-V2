### Product promise and north star

**Promise:** an adult in India can quickly receive one culturally, financially and personally
plausible complete outfit, understand why it was chosen, correct it, act on it and see the next
decision improve—free, private, accessible and truthful.

**North star:** weekly trusted outfit deciders: distinct consented users who save an explained
complete outfit joined to a real served recommendation ID, with reversals preserved. Activation is
the first such save in the first session. D7 and D30 retention require another save, correction or
wardrobe decision. Existing frozen GO/HOLD/fail thresholds and sample floors remain binding.

### Phase packet and evidence gate

Before a phase starts, the contract must name its user job, entry state, exit state, in-scope write
set, exclusions, dependencies, risks, acceptance tests, deployed proof, budgets, rollback and deletion
eligibility. During execution there is one in-progress phase and one root-cause hypothesis. After
execution, an independent review must reproduce the evidence. A file, route, passing mock, local
screenshot or model output is never sufficient proof by itself.

Gate outcomes are mechanical:

- **GO:** every required check and deployed outcome passes; record evidence and advance once.
- **HOLD:** evidence is incomplete or statistically inconclusive; remain in phase and gather only
  the missing evidence.
- **FAIL:** a floor regresses or the hypothesis is falsified; rollback, identify the root cause and
  rerun the same gate.
- **OWNER DECISION:** required only for spend step-up, production topology change, destructive
  deletion outside an already-approved group, changed product scope or public launch.

#### Immaculate experience direction: GYF Atelier / Cosmos Editorial

The visual goal is a quiet editorial atelier around vivid fashion—not a generic dashboard, glass
demo or dense commerce grid. Product imagery and the complete outfit are the hero; interface chrome
is calm, precise and subordinate. The differentiating interaction is an **evidence rail**: every
outfit exposes context, matched constraints, uncertainty and the fastest correction beside the
look, then visibly carries that correction into the next recommendation.

The design system has one token source and one component source:

- **Typography:** Fraunces only for editorial display moments; Bricolage Grotesque for navigation,
  labels and actions; system sans for long body copy. Two branded faces maximum, dynamic type
  supported, no text baked into images and no action truncated at largest supported scale.
- **Colour:** accessible ink/porcelain light and dark foundations; one owner-approved signature
  accent family; semantic success/warning/error tokens never reused as decoration. Outfit imagery
  supplies visual richness. Every text/control/state pair is programmatically contrast-tested;
  colour is never the sole state cue.
- **Shape:** continuous curves for hero frames, sheets and primary controls; restraint elsewhere.
  One distinctive orbital/curved outfit frame is preferable to rounding every container.
- **Spacing:** existing compact/phone/regular/wide tiers remain the only width system. Use flex and
  `useWindowDimensions`; no device-name branching or scattered numeric breakpoints.
- **Elevation/material:** flat content by default; blur/glass only for transient navigation or
  sheets where contrast survives. No nested translucent cards, fake depth or platform-inconsistent
  shadows. Android/web receive an opaque accessible equivalent.
- **Content:** real outfit/product/user states only. Loading, empty, partial, stale, offline,
  permission-denied, abstained, error, retry, end-of-catalogue and destructive-confirmation states
  are first-class art-directed fixtures, not afterthoughts.

P3 begins with four owner-reviewable route compositions—first recommendation, Explore, outfit
detail/correction and the manual Personal Fit onboarding path—across compact Android, regular
Android and responsive web, light and dark. The owner approves one direction before route-wide
composition. Route acceptance then covers every state; a pretty hero screenshot cannot promote the
system.

#### Navigation and information architecture

The stable top-level mental model remains five destinations: **Stylist, Explore, Wardrobe, Social,
Profile**. Stylist owns the decision loop. Saved, Collections, Account, Status, Support, Grievance,
Try-on and item/outfit detail are stack destinations, sheets or profile/account children—not extra
tabs. Tabs never appear/disappear by feature flag or auth state.

Each tab has a native stack title, deep-linkable URL, deterministic back behavior, restoration
state and route-local loading/error boundary. Shared item/outfit/profile detail routes can be pushed
from more than one tab without duplicating screens. Modals and form sheets use Router presentation,
not hand-built overlay navigation. Core gestures always have visible buttons. Web receives keyboard,
focus, hover and direct-link behavior; native receives safe insets, back gesture/back button and
screen-reader focus restoration.

#### Client state, networking and offline behavior

Keep the existing typed `GyfApi` as the single transport and move it to `expo/fetch` only through a
contract-preserving adapter with cancellation, request ID, timeout, parsed typed error and valid
status/content checks. No Axios. `EXPO_PUBLIC_*` contains only public build-time configuration;
secrets never enter a client bundle. Supabase session material stays in SecureStore on native and
the approved secure web session mechanism; refresh is single-flight and sign-out clears protected
cache/state.

Adopt TanStack Query v5 only as a bounded P2/P3 server-state simplification if the before/after
inventory proves it deletes duplicated loading/cache/refetch/race code. If adopted:

- query keys are typed by resource/user/filters; the authenticated user boundary is always part of
  protected cache ownership;
- request signals propagate to the transport; navigation/unmount/search replacement cancels work;
- retries are capped and only for safe transient reads—never auth, validation, destructive actions
  or arbitrary 4xx; exponential backoff honors server rate limits;
- mutations use server idempotency/event IDs, explicit optimistic snapshots and rollback; no
  optimistic account deletion, purchase claim, upload completion or ML result;
- invalidation targets exact affected resources; no global refetch storm;
- stale time follows truth: profile/consent relatively stable, catalogue availability short,
  recommendations user-action driven, system status live;
- offline mode reads clearly labelled cached public/non-sensitive data. Sensitive mutation queues
  are off by default; feedback/save queues require stable IDs, consent, encryption and reversal.

Do not add Redux, Zustand, MobX or a client database for ordinary screen state. Local form and
interaction state stays local; URL/search params own shareable navigation state; server truth stays
in the query layer. Add SQLite only if a measured offline wardrobe requirement passes a separate
privacy/encryption/sync-conflict design.

#### Backend and API architecture

Keep one typed FastAPI modular monolith. Routers validate transport and delegate; domain modules own
policy; repositories own SQL; capability ports isolate catalogue sources, encoders, rankers, event
sinks and try-on renderers. Pydantic contracts remain the boundary; client/shared types are generated
or proven against OpenAPI in CI. A schema change includes compatibility classification, consumer
tests, migration order, rollback and removal date. No versioned `/v2` fork without a real breaking
consumer need.

Keep psycopg pooling and explicit SQL for hot paths; SQLAlchemy/Alembic remains migration tooling,
not a second runtime data model. Every request has a deadline; database statements have bounded
timeouts; connection-pool wait is measured; external calls have connect/read/total timeouts,
bounded retries with jitter, circuit/kill behavior and deterministic fallback where truthful.
No catch-all fallback may turn an invalid/empty/error response into apparently valid content.

Use PostgreSQL `FOR UPDATE SKIP LOCKED` job tables for bounded asynchronous work, with idempotency,
leases, attempt caps, cancellation, dead-letter state, TTL deletion and per-user/global quotas.
Do not add Redis, Celery, Kafka, Redpanda, Kubernetes or microservices until measured concurrency,
durability or independent-scaling evidence exceeds this design and the cost/operability scorecard
approves a replacement.

#### Data, catalogue and learning contracts

PostgreSQL is the source of truth; RLS/owner scope is defense in depth, not a substitute for API
authorization. Migrations are forward-safe, one-head, lock/time-budgeted and rehearsed against
production-like cardinality. Restore evidence—not backup configuration—closes durability.

Catalogue rows require source/provider, licence/use rights, retailer identity, canonical URL,
currency/price, availability, freshness, category/taxonomy, image provenance and removal state.
Ingestion is idempotent and reconciled; dedupe preserves references; unavailable products remain
renderable in saved/wardrobe history but not newly recommended as purchasable. Training rights are
separate from listing/display rights.

Every served recommendation/item exposure gets a stable recommendation/event identity. Save,
skip, correct, shop, follow, react, share, try-on and undo/reversal events join to what was actually
served. Consent purpose, deletion tombstone and lineage travel into exports and training sets.
No model trains while join integrity is below 99%, rights are unknown, or temporal availability
cannot be reconstructed.

#### Recommendation, perception and VTON lifecycle

The production incumbent remains simple and inspectable: constraint filtering → SigLIP-compatible
retrieval → category-aware outfit compatibility/composition → rule/taste scoring → MMR diversity →
fact-derived explanation/confidence/abstention. Cache only versioned pure stages; a model/catalogue/
policy version change invalidates by construction. Every stage emits bounded timing and a reason for
empty/abstained output.

The first learned challenger is pairwise logistic/BPR-scale, trained only on consented, joined,
temporally valid outcomes. Offline evaluation uses global-time splits, candidates available at
ranking time, Recall/NDCG, diversity/repetition, constraint satisfaction, calibration and subgroup
slices. Shadow and cohort evaluation then require a lift in trusted outfit decisions with latency,
fairness, privacy, cost and correction rate non-inferior. Large sequential/generative/GNN systems,
feature stores and online learning remain rejected until a small baseline plus data-volume analysis
proves the ceiling.

Photo assistance is optional and correctable. Manual body/tone/undertone remains truth. Candidate
models report coverage, error/calibration by subgroup, abstention and correction impact; no beauty,
health, ethnicity or fit certainty is inferred. Raw media is private, EXIF-stripped, purpose-limited,
TTL-deleted and excluded from training without separate explicit authorisation.

VTON stays asynchronous behind `TryOnRenderer`: private input/output, durable job state, capped
attempts/concurrency/quota, cancellation, TTL erasure, cost accounting and global kill switch.
FASHN v1.5 is a candidate, not a claim of SOTA or size/fit truth. Hash/licence all transitive
artifacts; score a rights-clean diverse panel; expose known failure and “visualisation” language.

#### Security, privacy and abuse resistance

- Supabase Auth owns identity; API verifies issuer/audience/signature/expiry and derives user scope
  from the token, never a client-supplied owner ID.
- SecureStore protects native session secrets; no token, private URL, model/provider key or database
  credential appears in logs, analytics, screenshots or `EXPO_PUBLIC_*`.
- Validate content type by decoded content, not filename; cap dimensions/bytes; strip metadata;
  store sensitive media privately with short-lived access and complete erasure lineage.
- Rate limits, quotas and idempotency cover auth, recommendation, feedback, uploads, social, support
  and VTON. Social adds report/block/moderation, ownership, anti-spam and safe share/deep links before
  opening.
- Consent is purpose-specific and reversible; export/erasure/session revocation are deployed real-
  database journeys. Logs/traces use allow-listed fields and never raw photos, free text, tokens or
  full sensitive prompts.
- Dependency, container and secret scans run in CI; threat modelling covers auth boundary, cross-user
  access, poisoned catalogue/events, SSRF/upload bombs, prompt/model abuse, affiliate redirect and
  operator privilege. Critical/high unresolved findings block promotion.

## F8 design record: the durable try-on spine (researched 2026-07-15, shipped)

Four parallel research passes (system design, backend, frontend, product design) preceded
the code. What follows is the binding record of what was decided and, more importantly,
*what was rejected and why* — so the next slice does not relitigate it, and so the
ceilings are stated instead of discovered.

### The shape: a job, not a request

A render takes 10-60s on a GPU. The synchronous `POST /tryon` therefore could not survive
its own success: Render's proxy kills the request, a deploy drops it, the client cannot
recover it, nothing retries, nothing bounds the cost, and the body photo sits in memory
for the whole call. **The queue-and-poll shape is not a UX preference — it is the only
shape this budget supports**, because scale-to-zero GPU means the first render after idle
is cold. The product must therefore promise "we'll have it shortly, and you can walk
away", never "instant". The durable half is the feature: *close the page, the render is
waiting when you come back.*

### Decisions, with the rejected alternative

| Decision | Rejected, and why |
| --- | --- |
| **Claim with `FOR UPDATE SKIP LOCKED`** | Advisory locks / `LISTEN-NOTIFY`: Supabase's transaction pooler does not support `LISTEN`, and NOTIFY buys nothing for a 30s job. SKIP LOCKED is also what makes a Render zero-downtime deploy (old + new instance live at once) safe *by construction* rather than by luck. |
| **Two drainers: in-process thread (latency) + GitHub Actions cron (durability)** | A separate Render worker: +₹600/mo to babysit work that is ~100% network wait. A Render cron: 1-minute granularity on top of a 60s render. GH Actions *alone*: 5-10min granularity, unusable as the interactive path. Running both is safe precisely because of SKIP LOCKED, and needs no coordination. |
| **Photo + render as `BYTEA` in the job row** | Supabase Storage: needs `GYF_SUPABASE_SERVICE_ROLE_KEY`, **which is recorded as exposed and unrotated** — F8 will not stand up a new trust boundary on a key that must be rotated first. The ceiling is stated in migration `0018`: safe only because TTL (24h) × daily cap (200) bounds resident bytes to ~50MB. **Trigger to move to R2: TTL past ~72h or cap past ~700.** |
| **The render is served by a route, never inlined in JSON** | base64 in the poll response: every "still working" poll would move megabytes. The route also makes TTL expiry a real 404 instead of a stale blob in React state. |
| **Quota derived by `COUNT(*)` over the jobs table** | A counter table/Redis counter: a second source of truth that drifts from reality after a rollback or a purge. The jobs table *is* the ledger, and erasure takes the quota with it for free. |
| **Cancel is honest about the GPU** | Claiming a running job is "cancelled" and refunded. It is not: the vendor's HTTP call is not interruptible, so a pre-claim cancel is genuinely free (and refunds) while a mid-render cancel stops the *wait*, not the *spend*. Neither the UI nor the docs may say otherwise. |
| **Abstention is a terminal success, never a retry** | Retrying it: the renderer looked and honestly declined (D6). Re-asking burns GPU to get the same "no". |
| **The UI fails CLOSED on the capability check** | The existing `useCapability` fails *open*, which is right when the fallback is a manual form — and wrong here, where the ask is a photo of the user's body. A `/system/status` blip must never be the reason GYF solicits one. `useCapabilityStrict` added. |
| **No invented progress** | A percentage bar the server never sent. The product's entire thesis is that it does not lie to users; that does not carve out an exception for chrome. Honest stage text, and a real bar *only* if a lane ever streams true step counts. |
| **The user's own photo is the waiting state** | A spinner, or worse a shimmering skeleton over a picture of someone's body. Waiting to see your own body judged is an intimate, anxious moment; the photo stays on screen, dimmed, visibly still theirs. |

### Ceilings — stated, not discovered later

1. **F8 depends on the F2.5 Render Starter flip.** On the free tier the instance sleeps, so
   the in-process drainer sleeps and latency is bounded only by the cron. Jobs stay durable
   (nothing is lost); the experience is not acceptable until that flip.
2. **The kill switch is load-bearing, not a safety net.** 100 users × 3 renders/day × 30 is
   9,000 renders/month against a cap of a few thousand. The break-even is
   `DAU% × daily_quota × 30 × users ≤ cap` — a number the owner needs *before* F11's beta,
   not after the bill.
3. **Mid-render cancellation cannot refund GPU seconds.** Only pre-claim cancels are free.
4. **Storage, not GPU, becomes the binding constraint** if the TTL or cap is raised past the
   thresholds in `0018`.
5. **Serving lane correction (evidence, not preference):** F2.5 measured ZeroGPU's cold start
   at 29.7s and the quota is recorded as dead; the text lane was already migrated off it for
   exactly this reason. The pinned FASHN candidate and every rights-clean GYF challenger follow
   the proven `ml/serving/modal_encoder.py` pattern (scale-to-zero, weights in a volume,
   one-model allow-list) rather than depending on ZeroGPU. F9 selects and promotes the lane;
   F8 packages candidates but opens none to users.

## Previous handoff: F5 — free recommendation incumbent (F4 gate closed 2026-07-14)
